## Verdict

untrusted path reaching a write sink.

## Findings
### SEC-1 · nit · Corrupt phase file is now a hard crash (deliberate availability trade-off)
**Where:** `src/state/store.ts:202-215` (`readPhaseFile` → `CorruptPhaseError`)

Before this range a `.mini/phases/phase-N.json` that failed `JSON.parse`
degraded silently to `null` (the bare header summary). It now throws
`CorruptPhaseError`. An attacker who can write `.mini/phases/*.json` — e.g. a
poisoned pull, or a git merge that left conflict markers in the file — can make
every command that loads that phase fail. That is a denial-of-service in the
narrow sense, **but** the attacker already controls the repo, and the change is
the correct security call: the old silent-`null` behaviour dropped
`goal`/`steps`/`humanNotes`/`autoCommit` and, on the next save, made that loss
permanent (silent data corruption). Failing loud beats silently rewriting state
from a truncated file. Note only — no change recommended; the trade-off is the
right way round. The ENOENT-stays-`null` split is correct: a genuinely missing
file is still benign.

## Checked and clean
- **Process execution** — `adversarialProject.ts:24` adds exactly one entry to
  the Claude allow-list: `Bash(mini findings list:*)`. It is a read-only `mini`
  subcommand and sits beside the already-present `mini findings add`. No new
  process is spawned in this range, nothing shells out with `shell:true`, and no
  `.mini/` content is interpolated into an argv. The allow-list is a *narrowing*
  of what the session may run, not a widening of injected input.
- **Filesystem / path traversal** — the one new write-adjacent guard,
  `assertGraphPathInside` (`buildGraph.ts:159-174`), is called at
  `writeGraphLayout` (`buildGraph.ts:191`) **before** `file.path` is joined into
  `graphFileAbs`/`tmpFileAbs`. It rejects empty, absolute (platform *and*
  posix), bare `..`, and `../`-prefixed normalized paths, so a collector that
  ever yielded an escaping path fails loudly instead of writing outside
  `.mini/graph/`. This is defense-in-depth — `collectFiles` (git ls-files / walk
  under `cwd`) cannot currently produce such a path — and it strictly reduces
  the traversal surface. It does not, and does not claim to, cover symlinked
  path components (pre-existing, out of range). No other read/write path changed.
- **Untrusted parsing** — `stripFindingsSections` (`runReport.ts:117-152`)
  parses the untrusted run-report body line by line. The heading regex
  `/^(#{1,6})[ \t]+(.*?)[ \t]*$/` runs per single split line, so the lazy
  `.*?` + trailing `[ \t]*$` is at worst O(n²) on one line's length — not
  exponential, and report lines are bounded; no realistic ReDoS for a local CLI.
  Matching is a literal case-insensitive title compare against a fixed two-entry
  allow-list (`adversarial findings`, `verify findings`), so it cannot delete an
  unrelated `##` section. On a clean body it is a no-op. `store.ts` still does
  `JSON.parse(raw) as Phase` with no schema validation, but that cast is
  pre-existing and unchanged here; this range only improved the surrounding
  error handling.
- **Prompt injection / agent-trust** — `sessionContext.ts:349-352, 441-444,
  519-523` now run the report body through `stripFindingsSections` before
  inlining it as `# Implementation report`. This *reduces* the injection surface:
  stale `## Adversarial findings`/`## Verify findings` text no longer
  accumulates into the prompt across re-runs. The human checkpoint and scoped
  tool sets are unchanged. The doc/prompt edits (`commands.ts`,
  `sessionContext.ts` security-out-of-scope text, `docs/**`) are guidance text
  steering the human toward running `mini security` as a separate pass — no new
  trust boundary is crossed.
- **Dependency surface** — `package.json` changed only the `version`
  (1.21.3 → 1.22.3). No dependency added, removed or bumped; no install-time
  script introduced.
