- [ ] store.ts loadPhase/readPhaseFile swallows every error (ENOENT and malformed JSON alike) -> when currentPhaseId is set but the phase file is corrupt, verify/adversarial print a misleading 'no phase' message and steer to /mini:next. Distinguish missing from corrupt and report the real problem.
- [x] resolveRange (src/range.ts) — asymmetric range start: fromSha reads preSha(fromPhase) with no fallback, so a range starting at the first phase (no autoCommit.preSha — phase 1 structurally can't have one) hard-fails instead of diffing from the project genesis. Fix: when fromPhase is the project's first phase (or its preSha is absent because nothing precedes it), fall back to the git empty-tree sentinel so the diff starts from the beginning; for other missing-preSha phases keep a clear error. Note: phases 1-13 also lack preSha and their early commits are squashed into the init commit, so ranges inside 1-13 stay inherently lossy — reliable boundaries exist from phase 14 onward plus the 'from genesis' case.
- [ ] adversarial-project: the session prompt tells the reviewer to run 'mini findings list' first for dedup (sessionContext.ts:659), but ADVERSARIAL_PROJECT_ALLOWED_TOOLS (src/commands/adversarialProject.ts) only allows 'Bash(mini findings add:*)', not 'mini findings list'. So the dedup step is instructed but not permitted -> in the terminal session it triggers a permission prompt / denial and dedup effectively doesn't run. Fix: add 'Bash(mini findings list:*)' to the allowed tools. Verified: the single-phase adversarial (adversarial.ts) does NOT have this contradiction because its prompt never instructs 'mini findings list' (no dedup step) — out of scope here, though adding dedup there is a separate possible improvement.
- [ ] adversarial-project: make the security pass harder to forget. Today the prompt marks security out of scope and only optionally suggests '/security-review' (sessionContext.ts:678-681), so it silently never runs unless the human remembers. Add an explicit, visible reminder at the END of the adversarial-project flow (e.g. the status line / final output) telling the human to run '/security-review' as a separate pass. Keep it a reminder only — do NOT auto-run it and do NOT file security findings into mini findings (security stays a separate output by design). Decide whether the reminder lives in the prompt's closing status block, the slash-command body, or both.

Value mini adds is range-scoping + durable cross-session output, NOT better analysis than Anthropic's. Multi-phase feature — plan via /mini:next.
- [ ] Security SEC-1 (docs): document the prompt-injection / agent-trust boundary. mini assembles prompts from git-shared .mini/ content (project.md, phase goals, run/discuss reports) and in auto/acceptEdits mode feeds them to a Claude session that can edit files + run allowed tools. A cloned/pulled repo with poisoned .mini/ is a prompt-injection vector against whoever runs 'mini auto'. Add a short Security section to README/docs: treat untrusted .mini/ as untrusted, don't run 'mini auto --yolo' on un-reviewed repos, the human 'done' checkpoint is the main safety. See .mini/security/range-1-25.md (SEC-1).
